# X 推荐内容抓取与归档平台需求文档（初稿）

## 1. 文档说明

- 文档状态：更新中
- 更新时间：2026-03-19
- 项目名称：auto-x-to-wechat
- 文档目标：明确平台第一阶段的产品需求、功能范围、技术边界与实施基础

## 2. 项目背景

本项目计划开发一个 Web 软件，允许用户在平台内登录并绑定自己的 x.com 账号。系统在绑定完成后，能够按设定周期抓取该账号推荐流中的帖子内容，对抓取结果进行去重与归档，并以卡片列表的形式在后台页面中进行分页展示，供用户后续检索、查看与进一步处理。

该项目的第一阶段重点是完成“账号绑定 -> 定时抓取 -> 去重存储 -> 富文本归档 -> 列表展示”这一完整闭环。

## 3. 产品目标

### 3.1 核心目标

1. 支持用户登录平台并绑定自己的 X 账号。
2. 支持系统按固定周期自动抓取该账号推荐流中的帖子。
3. 对已抓取帖子按帖子 ID 去重，避免重复入库。
4. 将帖子内容保存为可展示的富文本归档内容。
5. 在系统中以卡片列表形式展示归档帖子，并支持分页浏览。

### 3.2 阶段性目标（MVP）

1. 单个平台注册用户支持绑定 1 个 X 账号。
2. 支持设置统一的抓取周期，或为账号配置独立抓取周期。
3. 支持手动触发一次抓取任务。
4. 支持查看抓取任务执行记录和结果概览。
5. 支持查看帖子列表和帖子详情页。

### 3.3 非目标

当前阶段不包含以下能力：

1. 不实现自动发布到微信公众号或其他第三方平台。
2. 不实现复杂的内容编辑工作流。
3. 不实现多成员协作、组织空间、审批流。
4. 不实现高级推荐算法分析或内容打分系统。
5. 不以对外开放 API 为第一阶段交付目标。

## 4. 用户角色

### 4.1 普通用户

普通用户是平台的主要使用者，能够完成：

1. 登录平台。
2. 通过浏览器辅助登录绑定、查看、解绑自己的 X 账号。
3. 配置抓取周期。
4. 查看抓取记录。
5. 浏览归档帖子列表和帖子详情。

### 4.2 系统管理员（预留）

第一阶段不强制要求实现完整后台管理端，但建议在数据模型和权限设计中预留管理员角色，以支持后续查看系统任务状态、失败日志、账号异常和全局数据统计。

## 5. 业务范围与核心流程

### 5.1 核心业务流程

1. 用户注册或登录平台。
2. 用户在平台中发起浏览器辅助绑定流程。
3. 系统验证绑定状态可用并自动回填登录账号信息。
4. 系统根据定时策略执行抓取任务。
5. 系统拉取该账号推荐流中的帖子数据。
6. 系统记录本次抓取任务及抓取到的帖子 ID。
7. 系统判断帖子 ID 是否已归档。
8. 若未归档，则解析帖子内容并保存为富文本归档。
9. 用户在前端页面中以卡片列表分页浏览归档内容。
10. 用户可点击查看帖子详情与原始来源信息。

### 5.2 去重范围

第一阶段默认按“绑定账号 + 帖子 ID”进行去重，即：

1. 同一绑定 X 账号下，同一个帖子只能归档一次。
2. 每次抓取任务都需要记录抓取到的帖子 ID 和处理结果。
3. 若未来允许多个平台用户绑定同一个 X 账号，需要进一步确认是按“全局帖子 ID 去重”还是“用户维度去重”。

## 6. 功能需求

### 6.1 登录与权限

#### 6.1.1 登录方式

1. 平台采用 NextAuth.js 实现身份认证。
2. 第一阶段建议支持至少一种稳定登录方式，例如邮箱验证码登录或 OAuth 登录。
3. 登录后才允许访问账号绑定、抓取配置、归档内容等页面。

#### 6.1.2 权限要求

1. 用户只能访问和管理自己的绑定账号、抓取任务和归档内容。
2. 未登录用户访问受保护页面时，应跳转到登录页。
3. 后端接口必须基于用户身份进行数据隔离。

### 6.2 X 账号绑定

#### 6.2.1 绑定能力

1. 用户可在平台中发起一个 X 账号浏览器辅助绑定流程。
2. 平台能够自动拉起浏览器并跳转到 `x.com` 登录页。
3. 用户在浏览器窗口中手动完成登录后，系统自动采集账号信息与登录态 Cookie。
4. 绑定成功后，系统需展示绑定状态，例如“已绑定”“失效”“待验证”。
5. 用户可主动解绑账号。
6. 用户可在绑定失效时重新绑定。

#### 6.2.2 绑定信息

至少需要保存以下信息：

1. X 用户 ID。
2. X 用户名（username）。
3. 显示名称。
4. 头像地址。
5. 绑定状态。
6. 用于抓取推荐流所需的授权信息或会话凭证。
7. 最近一次验证时间。
8. 最近一次浏览器绑定会话状态与错误摘要。

#### 6.2.3 安全要求

1. 与 X 账号绑定相关的敏感凭证必须加密存储。
2. 前端不得直接暴露敏感凭证。
3. 后端日志中不得打印完整 token、cookie 或会话信息。

#### 6.2.4 待确认事项

由于“抓取推荐流”通常依赖登录态，且不一定能通过官方开放接口直接获取，第一阶段明确采用“平台拉起受控浏览器 -> 用户手动登录 -> 系统自动采集登录态 -> 保存绑定并执行抓取”的方案。后续如需远程部署，可再演进为远程浏览器或扩展采集方案。

### 6.3 定时抓取任务

#### 6.3.1 抓取配置

1. 用户可为绑定账号启用或关闭自动抓取。
2. 用户可设置抓取周期，例如每 15 分钟、30 分钟、1 小时、2 小时。
3. 系统需记录下一次计划执行时间。
4. 系统需支持手动立即抓取一次。

#### 6.3.2 抓取目标

1. 抓取目标为该 X 账号登录后首页推荐流中的帖子内容。
2. 第一阶段以文本内容和基础元数据抓取为主。
3. 若帖子包含图片、视频、引用帖、转发帖，系统应尽量保留结构化信息或原始引用关系。
4. 第一阶段真实抓取基于浏览器自动化打开 `x.com/home` 并抓取可见推荐流内容。

#### 6.3.3 抓取结果记录

每次抓取任务需记录：

1. 抓取任务 ID。
2. 触发方式（定时/手动）。
3. 任务开始时间与结束时间。
4. 任务状态（成功、部分成功、失败）。
5. 抓取到的帖子总数。
6. 新增归档数量。
7. 跳过数量（已存在或无效数据）。
8. 失败原因摘要。

#### 6.3.4 异常处理

1. 若账号凭证失效，应将账号状态标记为“失效”，并提示用户重新绑定。
2. 若抓取过程中出现网络异常、限流或页面结构变化，应记录失败日志。
3. 任务失败不应影响历史归档数据。
4. 对同一账号的重复执行任务需有并发控制，避免同一时刻重复抓取。

### 6.4 帖子去重与归档

#### 6.4.1 去重规则

1. 系统以帖子 ID 作为主要唯一标识。
2. 针对同一绑定账号，若帖子 ID 已存在，则本次抓取不重复归档。
3. 每次抓取仍需记录“该帖子已抓取过并被跳过”的处理结果。

#### 6.4.2 归档内容范围

系统归档时，至少保存以下字段：

1. 帖子 ID。
2. 帖子链接。
3. 作者信息。
4. 原始文本内容。
5. 解析后的富文本内容。
6. 发布时间。
7. 抓取时间。
8. 媒体资源信息（如有）。
9. 帖子类型（普通帖、转发、引用帖、回复，若可识别）。
10. 原始抓取载荷快照或必要的原始 JSON 数据。

#### 6.4.3 富文本归档要求

1. 系统需要将帖子内容转换为适合前端展示的富文本格式。
2. 富文本需尽量保留以下内容：段落与换行、链接、提及账号、话题标签、图片或视频的引用位置、引用帖或转发帖的关系说明。
3. 建议同时保存一份结构化 JSON 和一份渲染后的 HTML 或可序列化富文本数据，以兼顾展示与后续扩展。

#### 6.4.4 数据一致性

1. 帖子归档写入与抓取记录写入需在逻辑上保持一致。
2. 归档成功后再将帖子标记为已处理。
3. 若归档过程中失败，应保留失败状态，允许后续补偿处理。

### 6.5 归档内容展示

#### 6.5.1 列表页

系统需提供帖子归档列表页，采用卡片列表展示，支持：

1. 分页浏览。
2. 每条卡片展示作者、发布时间、摘要内容、媒体缩略图和来源链接。
3. 展示归档时间。
4. 展示帖子类型标签。
5. 点击进入详情页。

#### 6.5.2 详情页

帖子详情页至少展示：

1. 完整富文本内容。

### 6.6 页面体验与国际化

#### 6.6.1 页面内容表达

1. 首页及关键页面文案应面向最终用户，优先介绍产品功能、使用价值与系统能力。
2. 页面中不应出现“当前阶段”“MVP 准备中”“推进路径”等偏研发过程的说明性文案。
3. 首页应突出真实 X 绑定、自动抓取、去重归档、富文本展示、抓取记录追踪等核心能力。
4. 首页可适度展示技术栈与部署方式，用于增强产品可信度和可维护性表达。

#### 6.6.2 Header 与导航体验

1. Header 需拆分为品牌区、功能导航区、工具区，避免导航项与切换控件拥挤在同一行。
2. 主导航需始终显示当前页面的选中状态，例如保留底色、高亮文字或其它持续可见的视觉反馈。
3. 语言切换与主题切换应归入工具区，不应影响主导航的可读性与点击区域。

#### 6.6.3 中英文国际化

1. 系统支持简体中文与英文两套界面语言。
2. 用户点击语言切换后，应在当前页面即时生效。
3. 语言偏好需在刷新页面、重新访问系统后继续保留。
4. 页面标题、按钮、提示语、状态文案、分页信息和空状态文案都需随语言切换同步更新。

#### 6.6.4 亮色与暗色主题

1. 系统支持亮色主题与暗色主题切换。
2. 主题切换应即时生效，并在后续访问中保留用户选择。
3. Header、页面背景、卡片、空状态、错误状态、导航项和主要按钮需适配双主题。
2. 作者信息。
3. 原始发布时间。
4. 归档时间。
5. 媒体资源。
6. 原始链接。
7. 所属抓取任务信息。

#### 6.5.3 列表筛选（建议）

第一阶段建议预留以下筛选条件：

1. 按绑定账号筛选。
2. 按抓取时间范围筛选。
3. 按帖子类型筛选。
4. 按关键词搜索。

若开发周期有限，可将筛选能力列为次优先级功能。

### 6.6 抓取记录展示

系统需提供抓取任务记录页面，至少包括：

1. 任务执行时间。
2. 触发方式。
3. 任务状态。
4. 抓取总数。
5. 新增数。
6. 跳过数。
7. 错误摘要。

用户可点击单次任务查看更详细的处理结果。

## 9. 接口与任务设计建议

### 9.1 前端应用

前端采用以下技术：

1. Next.js
2. TypeScript
3. Tailwind CSS
4. ShadCN UI
5. NextAuth.js

前端主要负责：

1. 登录与会话管理。
2. 账号绑定页面交互。
3. 抓取任务配置与触发。
4. 归档内容列表与详情展示。

### 9.2 后端服务

后端采用以下技术：

1. NestJS
2. Prisma
3. PostgreSQL

后端主要负责：

1. 用户业务数据管理。
2. X 账号绑定信息管理。
3. 抓取任务调度与执行。
4. 帖子解析、去重与归档。
5. 为前端提供 REST API 或 GraphQL API。

### 9.3 定时任务建议

1. 建议由 NestJS 定时任务模块或队列系统执行抓取任务。
2. 若后期抓取量增大，建议引入队列，例如 BullMQ。
3. 调度器应支持失败重试、超时控制和并发限制。

## 10. 非功能需求

### 10.1 安全性

1. 敏感凭证加密存储。
2. 服务端接口必须进行身份校验和权限校验。
3. 防止越权访问其他用户数据。
4. 支持基础操作审计日志。

### 10.2 性能

1. 归档列表页在常规分页场景下应快速返回。
2. 归档表需建立合理索引，至少包括 bindingId、xPostId、archivedAt。
3. 抓取任务不应阻塞用户正常访问。

### 10.3 可维护性

1. 前后端项目结构需清晰分层。
2. Prisma 数据模型需具备明确迁移方案。
3. 关键模块需补充基础测试。

### 10.4 可观测性

1. 记录抓取任务运行日志。
2. 记录失败原因和异常栈。
3. 为后续监控告警预留接口。

## 11. 部署要求

### 11.1 开发与版本管理

1. 代码使用 Git 进行版本管理。
2. 建议建立基础分支策略和提交规范。

### 11.2 部署方案

支持以下部署方式：

1. Docker 部署完整服务。
2. 前端部署至 Vercel。
3. PostgreSQL 使用 Neon 托管。

### 11.3 部署建议

1. 若抓取逻辑依赖浏览器自动化或长时间任务，需要确认 Vercel Serverless 是否满足执行需求。
2. 若不满足，建议将抓取服务独立为容器化后台服务，前端仍部署在 Vercel。
3. 数据库统一使用 PostgreSQL，保证前后端环境一致。

## 12. MVP 验收标准

当以下条件满足时，可认为第一阶段 MVP 基本完成：

1. 用户能够成功登录平台。
2. 用户能够绑定一个可用的 X 账号。
3. 系统能够按设定周期执行抓取任务。
4. 系统能够记录每次抓取结果。
5. 同一绑定账号下的帖子不会因重复抓取而重复入库。
6. 新抓取到的帖子会被解析并归档为可展示内容。
7. 用户能够在页面中以卡片列表查看归档帖子并进行分页浏览。
8. 用户能够查看某条帖子的详情内容。

## 13. 风险与待确认项

### 13.1 高优先级风险

1. X 推荐流抓取能力可能受登录方式、页面结构变化、限流与风控影响。
2. 若采用浏览器自动化，部署资源和稳定性成本会提高。
3. 若采用会话凭证存储方式，需要重点评估安全和合规边界。

### 13.2 待确认问题
