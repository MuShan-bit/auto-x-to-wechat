# 上线检查清单

## 1. 上线前确认

- [ ] 确认正式环境使用的 Git 提交版本、镜像版本、Vercel 部署版本一致。
- [ ] 确认 `apps/web` 与 `api` 已在预发或测试环境完成主流程回归。
- [ ] 确认本次发布涉及的数据结构变更、接口变更、环境变量变更已同步给相关成员。

## 2. 环境变量检查

### 2.1 Web / Vercel

- [ ] `NEXTAUTH_SECRET` 已配置。
- [ ] `NEXTAUTH_URL` 已配置为正式 HTTPS 域名，或确认可使用 Vercel 域名兜底逻辑。
- [ ] `AUTH_TRUST_HOST=true` 已配置。
- [ ] `DATABASE_URL` 已指向 Neon PostgreSQL。
- [ ] `INTERNAL_API_BASE_URL` 已指向正式 NestJS API。
- [ ] `INTERNAL_API_SHARED_SECRET` 已与 API 环境保持一致。

### 2.2 API / Docker

- [ ] `DATABASE_URL` 已指向正式 Neon PostgreSQL。
- [ ] `INTERNAL_API_SHARED_SECRET` 已配置。
- [ ] `CREDENTIAL_ENCRYPTION_KEY` 已配置，且长度与格式符合要求。
- [ ] `CRAWLER_ADAPTER_NAME` 已按正式策略确认。
- [ ] `X_BINDING_SESSION_TIMEOUT_SECONDS`、`REAL_CRAWLER_MAX_POSTS` 已确认。
- [ ] 若需真实浏览器抓取，已确认浏览器运行环境、可见性与相关路径配置。

## 3. 数据库迁移检查

- [ ] 已确认 Neon 备份、分支或 PITR 回滚能力可用。
- [ ] 已在正式环境变量下执行 `pnpm db:migrate:status`。
- [ ] 已在正式环境变量下执行 `pnpm db:migrate:deploy`。
- [ ] 已保存本次迁移输出结果，确认无失败或待处理迁移。
- [ ] 已确认迁移与当前 API 版本兼容，不会阻断回滚。

## 4. 发布执行顺序

- [ ] 先执行数据库迁移，再发布 API 容器。
- [ ] API 发布后已通过 `/health` 健康检查。
- [ ] API 发布后已验证关键接口：绑定列表、归档列表、抓取运行记录、仪表盘汇总。
- [ ] API 稳定后再发布或提升 Vercel 前端到正式版本。
- [ ] 前端发布后已验证登录、绑定、手动抓取、归档详情、分页浏览。

## 5. 日志与运行状态检查

- [ ] 检查 NestJS 结构化日志，确认无持续性 `FAILED` / `PARTIAL_FAILED` 异常。
- [ ] 检查 Dashboard 失败汇总，确认失败任务数量在预期范围内。
- [ ] 检查 Vercel Function 日志，确认认证与 BFF 路由无持续 5xx。
- [ ] 检查 Neon 连接数、慢查询、错误日志是否异常。
- [ ] 检查最近一次定时抓取是否成功落库且没有重复归档。

## 6. 回滚策略

- [ ] 已确认前端回滚方式：直接回退到上一个稳定的 Vercel Deployment。
- [ ] 已确认 API 回滚方式：回退到上一个稳定镜像或 Git 发布版本。
- [ ] 已确认数据库回滚方式：使用 Neon 备份、分支回切或 PITR。
- [ ] 若迁移不可逆，已准备兼容脚本或人工处置方案。
- [ ] 若抓取链路异常，已准备临时关闭抓取任务或解绑入口的应急方案。

## 7. 发布后 30 分钟观察

- [ ] 观察登录成功率、关键页面访问和 API 5xx 是否异常。
- [ ] 观察一次真实或测试抓取任务，确认结果可展示到归档页。
- [ ] 观察新增日志中是否出现认证失败、数据库连接失败、抓取结构变更错误。
- [ ] 记录本次发布时间、负责人、异常情况与处置结论。
