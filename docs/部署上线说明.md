# 部署上线说明

## 1. 本地 Docker 启动

### 1.1 启动 PostgreSQL + API

```bash
docker compose up --build db api
```

说明：

1. `api` 容器启动时会显式执行 `pnpm db:migrate:deploy`，确认迁移状态后再启动 NestJS 服务。
2. 默认暴露端口：`db=55432`、`api=3001`。
3. `docker-compose.yml` 默认将 `api` 服务固定到 `linux/amd64`，以便在 Apple Silicon 等环境中通过 QEMU 运行 `Google Chrome Stable`，提升 X / Google 交互登录成功率。
4. `docker-compose.yml` 默认将 `CRAWLER_ADAPTER_NAME` 设置为 `mock`，便于本地联调归档、列表、分页和任务调度链路。
5. 如需调整宿主机端口，可在启动前覆盖 `DOCKER_DB_PORT`、`DOCKER_API_PORT`。

### 1.2 启动完整服务

```bash
docker compose --profile full up --build
```

说明：

1. `full` profile 会额外启动 `web` 服务，暴露端口 `3000`。
2. `web` 容器通过 `INTERNAL_API_BASE_URL=http://api:3001` 访问 NestJS API。
3. `web` 运行时同样需要访问 PostgreSQL，以支撑 NextAuth Prisma Adapter。
4. 浏览器辅助绑定会通过 noVNC 暴露远程桌面，默认地址为 `http://localhost:6080/vnc.html?autoconnect=1&resize=scale&reconnect=1`。
5. 如需调整前端宿主机端口，可在启动前覆盖 `DOCKER_WEB_PORT`。

### 1.3 与真实 X 绑定的关系

当前容器编排已内置远程桌面链路，可直接在浏览器打开 noVNC 页面完成真实的 X 交互式绑定。为降低 Google 登录将容器浏览器识别为不安全环境的概率，默认方案是 `linux/amd64 API 容器 + Google Chrome Stable + noVNC`。如果你从其他机器访问服务，需要把 `X_BROWSER_REMOTE_DESKTOP_URL` 中的 `localhost` 改成实际宿主机 IP 或域名。

## 2. Vercel 前端部署

### 2.1 项目配置

1. 在 Vercel 中将 Frontend Project 的 Root Directory 设置为 `apps/web`。
2. 仓库内已提供 `apps/web/vercel.json`，固定使用 `Next.js` 框架和 `next build` 构建命令。
3. `src/app/api/**/*` 的 Vercel Function 超时时间已统一提升到 `30s`，便于前端 BFF 路由调用 NestJS API。
4. `apps/web/.vercelignore` 已排除测试、Playwright 结果和本地构建产物，减少部署上传体积。

### 2.2 环境变量

Vercel 中至少需要配置以下变量：

1. `NEXTAUTH_SECRET`
2. `AUTH_TRUST_HOST=true`
3. `DATABASE_URL`
4. `INTERNAL_API_BASE_URL`
5. `INTERNAL_API_SHARED_SECRET`

推荐按环境分别配置：

1. Preview：`INTERNAL_API_BASE_URL` 指向测试或预发 NestJS API。
2. Production：`INTERNAL_API_BASE_URL` 指向正式 NestJS API。
3. `DATABASE_URL` 需指向 Neon PostgreSQL，以支撑 NextAuth Prisma Adapter。

### 2.3 域名与回调地址

1. 若配置了正式自定义域名，建议显式设置 `NEXTAUTH_URL=https://你的正式域名`。
2. 若未显式设置 `NEXTAUTH_URL`，前端会自动回退使用 `VERCEL_PROJECT_PRODUCTION_URL`、`VERCEL_BRANCH_URL` 或 `VERCEL_URL` 推导站点地址，适配 Preview 与默认域名场景。
3. 若 API 有来源校验、WAF 或 IP 白名单，需要额外放通 Vercel 到 API 的服务端访问。

## 3. 生产数据库迁移流程

### 3.1 迁移命令

仓库根目录已提供以下命令：

1. `pnpm db:migrate:status`
2. `pnpm db:migrate:deploy`

其中 `pnpm db:migrate:deploy` 会先执行 `prisma migrate status`，再执行 `prisma migrate deploy`，适合作为生产发布前的显式迁移步骤。

### 3.2 推荐发布顺序

1. 准备好生产环境变量，至少包括 `DATABASE_URL`。
2. 先执行 `pnpm db:migrate:status`，确认目标库连接正常并检查待执行迁移。
3. 在发布新版本 API 之前，执行 `pnpm db:migrate:deploy`。
4. 待迁移成功后，再滚动发布新的 NestJS API 容器。
5. 发布完成后，通过健康检查与核心页面回归验证数据库读写是否正常。

### 3.3 容器化发布建议

1. API Docker 镜像默认仅负责启动服务，不再把“每次容器启动都自动迁移”当作生产流程。
2. 若走容器发布，建议把 `pnpm db:migrate:deploy` 配置为独立 release job、init job 或一次性 Job。
3. 若使用 Neon，请先在非生产库验证迁移，再执行正式库迁移。
